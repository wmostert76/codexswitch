# AGENTS.md

## Project

CodexSwitch Commander is a Python terminal application that configures Codex
CLI for native OpenAI or OpenCode Go models.

The repository contains:

- `bin/codexswitch`: CLI backend and launcher
- `bin/codexswitch-tui`: Textual Commander-style interface
- `bin/codex-opencode-go-proxy`: Responses API compatibility proxy
- `bin/opencode-go-token`: Reads the existing OpenCode Go credential
- `install.sh`: Reproducible local installation

## Safety

- Never commit authentication files, API keys, OAuth tokens or token output.
- Never print token values during tests.
- Treat `~/.codex/auth.json` and
  `~/.local/share/opencode/auth.json` as secrets.
- Preserve permissions `0700` for account directories and `0600` for account
  files.
- Do not delete or overwrite saved accounts unless explicitly requested.
- Before replacing `~/.codex/auth.json`, sync the active account back to the
  account store. OpenAI refresh tokens rotate during normal Codex usage.
- Avoid placing user-specific paths, account names or email addresses in code,
  documentation, screenshots or fixtures.

## Development

Use the project virtual environment for TUI work:

```bash
python3 -m venv .venv
.venv/bin/pip install -r requirements.txt
```

Keep the backend usable without Textual. `codexswitch classic` and all
non-interactive subcommands must continue to work with the system Python.

Prefer the existing backend functions over duplicating provider, account or
configuration logic in the TUI.

## Checks

Run before committing:

```bash
python3 -m py_compile bin/codexswitch bin/codex-opencode-go-proxy bin/opencode-go-token bin/codexswitch_common.py
.venv/bin/python -m py_compile bin/codexswitch-tui
bash -n install.sh uninstall.sh
.venv/bin/python -m pytest tests/ -v
codexswitch list
codexswitch status
```

For TUI changes, run a Textual headless test at both `80x24` and `120x40`.
Verify that both panes, the reasoning selector, status bar and all function-key
labels fit without wrapping or overlap.

For provider integration tests:

1. Save the current provider, model, reasoning mode and OpenAI account.
2. Test OpenCode Go through the normal `codex` command, not only the upstream
   API.
3. Restore the exact original configuration even when a test fails.

## Repository Hygiene

- Keep `.venv`, caches, logs and generated bytecode untracked.
- Keep screenshots sanitized with fictional account data.
- Update the README when commands, keys, installation or auth behavior changes.
- Do not weaken the root-owned installed command files merely to simplify local
  development; edit the repository and reinstall or retain symlinks.

## Known Issues

- **apply_patch tool via OpenCode Go proxy**: The proxy correctly converts
  custom tool calls (type `custom_tool_call`) to and from the upstream chat
  completions format, and unwraps/re-wraps the `{"input": "..."}` envelope.
  However, Codex CLI still rejects apply_patch payloads with
  "incompatible payload" when using OpenCode Go models (GLM-5.2, DeepSeek V4).
  This appears to be a Codex-side issue with how it validates custom tool call
  responses from non-OpenAI providers. The `exec_command` and `write_stdin`
  tools work correctly. This needs further investigation — check whether a
  newer Codex version fixes it, or whether the SSE event sequence for custom
  tool calls needs adjustment (e.g. `response.custom_tool_call.output_text.delta`
  instead of `response.function_call_arguments.delta`).
